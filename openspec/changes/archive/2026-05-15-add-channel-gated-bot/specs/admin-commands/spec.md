## 新增需求

### 需求:命令分发器必须通过首词匹配识别命令

系统必须识别以 `/` 开头的消息作为命令候选;必须按 `/<command>` 或 `/<command>@<bot_username>` 两种形式匹配,后者的 `<bot_username>` 必须匹配当前 bot 的 username(大小写不敏感),不匹配则必须忽略;命令关键字必须小写化后比较。文本首字符必须是 `/`;前导空白字符(空格、制表、换行)必须使消息**不被视为命令**,由 gating 按普通文本处理。分发器必须仅检查 `msg.Text`,媒体消息的 `caption` 永远不得被识别为命令触发,即便 caption 内容与某个已注册命令完全一致。系统必须只识别纯命令调用(`/cmd` 或 `/cmd@bot_username`,其后无任何实际文本);带有尾随文本的消息(如 `/bind foo`、`/status buy now`)必须不视为命令,而是由 gating 流水线作为普通用户消息处理。这样既关闭了以命令为前缀绕过删除的旁路,也让调度逻辑保持简单。`@` 后必须是合法的 Telegram bot username(5–32 位,`[A-Za-z0-9_]`,且首字符为字母);否则整条消息不被视为命令,由 gating 按普通文本处理。匿名管理员命令(`from.id == 1087968824` 且 `sender_chat.id == chat.id`)必须被 dispatcher 正常路由到对应 handler,由 handler 回复拒绝消息并清理。系统必须不跳过 dispatch,因为 `sender_chat` 指向本群本身属于用户身份的变体。注意:gating 流水线的命令短路(见 channel-gating 规格)比分发器的 `/<command>@<本 bot>` 匹配更宽——它还须覆盖指向其他 bot 的纯命令(`/cmd@其他 bot`),以便与其他 bot 共存;分发器本身仍只路由指向本 bot 的命令到 handler。

#### 场景:带 bot username 后缀的命令
- **当** 用户在群内发送 `/status@my_guardian_bot`,当前 bot username 为 `My_Guardian_Bot`
- **那么** 系统必须识别为 `/status` 命令并路由到对应 handler

#### 场景:指向其他 bot 的命令
- **当** 用户发送 `/status@other_bot`
- **那么** 系统必须忽略此消息

#### 场景:媒体 caption 与命令字样相同
- **当** 用户在群内发送一张图片,caption 为 `/status`,text 为空
- **那么** 分发器必须不路由至任何 handler;该消息必须走完整 gating 流水线

#### 场景:畸形 @ 后缀不被视为命令
- **当** 非成员发送 `/status@` 或 `/@other_bot` 或 `/spam@x`(username 非法或为空)
- **那么** 系统必须不视为命令,必须将该消息交给 gating 流水线作为普通文本处理;若发送者非频道成员必须删除

#### 场景:前导空白不被视为命令
- **当** 用户发送 " /status"、"\n/bind" 或其他首字符为空白的文本
- **那么** 系统必须不路由到命令 handler,也不启用命令短路;该消息由 gating 按普通文本处理,若发送者非频道成员必须删除

### 需求:/status 命令必须仅对群创建者可用,且输出绑定与运行状态

`/status` 命令必须在执行前调用 `getChatMember(group_id, caller_user_id)`,要求 `creator`(严格相等);非创建者必须收到拒绝回复;权限通过后系统必须在群内回复一条消息,包含:当前 `group_chat_id`、绑定的 `channel_chat_id` 与频道标题、`verified_members` 中该群的有效条目数、近 1 小时该群的删除消息计数、最近 5 次 `getChatMember` 错误的时间戳与错误信息。"已验证成员"计数必须仅统计当前生效绑定的 `channel_chat_id` 下的行,不得把历史其他频道残留的 `verified_members` 行计入。若 `CountVerifiedInChannel` 返回错误(例如 SQLite 锁冲突),系统必须回复"内部错误,读取状态失败"而非用 0 作为伪造数字继续渲染;原命令消息与错误回复按与其他拒绝路径一致的方式在 10 秒后被清理。当读取绑定频道元数据 (`GetChat(channel)`) 失败时,系统必须回复"内部错误,读取频道信息失败"而非用 "(未知)" 占位符继续渲染正常报告。状态报告回复失败(SendMessage 返错)时必须记 ERROR 日志并向 runtime 返回错误;不得伪装成功。与 /bind 成功后回复失败的行为保持一致。

#### 场景:创建者查询状态
- **当** 创建者 A 在群 G 内执行 `/status`,G 已绑定到频道 C
- **那么** 系统必须回复一条文本消息,包含字段:`group`、`channel`、`cached_members`、`deletes_1h`、`recent_errors`

#### 场景:非创建者查询状态
- **当** 非创建者(普通成员或管理员)在群 G 内执行 `/status`
- **那么** 系统必须回复权限不足提示,必须不泄露任何运行时状态字段

#### 场景:未绑定的群执行 /status
- **当** 创建者 A 在未绑定的群 G 执行 `/status`
- **那么** 系统必须回复"当前群未绑定任何频道",必须不查询 `verified_members` 或删除计数

#### 场景:/status 读取验证缓存计数失败
- **当** 创建者执行 `/status`,后端 `CountVerifiedInChannel` 返回错误
- **那么** 系统必须回复"内部错误,读取状态失败",不得输出带有 0 条验证成员的伪造状态块,必须清理原命令与错误回复

#### 场景:/status 读取频道元数据失败
- **当** 管理员执行 `/status`,绑定存在,但 `GetChat(channel)` 返回错误
- **那么** 系统必须回复"内部错误,读取频道信息失败",必须不输出带占位符的状态块,必须清理原命令与错误回复

### 需求:所有管理员命令必须返回 Markdown 格式的中文回复

`/bind`、`/unbind`、`/status` 的回复消息必须使用 Telegram `ParseMode = MarkdownV2`;回复文本必须使用简体中文;动态内容(频道标题、群名等)必须做 MarkdownV2 转义,防止特殊字符导致解析失败或注入格式。`/bind` 与 `/unbind` 的命令消息与回复在处理完成后必须按 group-binding 规格中的"/bind 与 /unbind 的命令消息与回复必须在 10 秒内自动删除"规定自动清理。

#### 场景:频道标题含 MarkdownV2 特殊字符
- **当** 频道标题为 `官方群 (测试)`,包含括号特殊字符
- **那么** `/bind` 回复中必须将括号转义为 `\(` 和 `\)`,Telegram 必须成功解析并展示

### 需求:/status 的拒绝性回复必须自动清理,成功状态报告必须保留

`/status` 的拒绝性回复(非群聊、非管理员、未绑定群、读取绑定失败、getChatMember 校验失败、带参数的非法调用等)必须与 `/bind` / `/unbind` 一致,在约 10 秒延迟后(而非立即)自动删除原命令消息与 bot 的回复,以便操作者在清理前能将拒绝回复与自己触发的命令关联起来。`/status` 成功的状态报告回复必须保留,不得自动删除——这是管理员需要阅读的内容,且是唯一不参与自动清理的命令路径。

#### 场景:/status 被非创建者拒绝后自动清理
- **当** 非创建者在群 G 执行 `/status`,被系统拒绝并回复"仅群创建者..."
- **那么** 系统必须在约 10 秒后删除原命令消息与拒绝回复

#### 场景:/status 带有尾随文本不被视为命令
- **当** 任何用户在群 G 执行 `/status foo bar` 或任何带尾随文本的变体
- **那么** 系统必须不调用 `/status` handler,必须不输出状态报告;此消息必须按普通消息由 gating 流水线处理,若发送者非频道成员,消息必须被删除

#### 场景:创建者 /status 成功响应不被清理
- **当** 创建者执行 `/status`,系统成功返回状态报告
- **那么** 系统必须不调度对原命令消息或状态报告回复的删除(这是唯一不清理的命令路径)

### 需求:纯命令消息必须不被当作受管控消息本身触发删除

`/bind`、`/unbind`、`/status` 等由 bot 识别处理的纯命令调用(`/cmd` 或 `/cmd@bot_username`,其后无任何实际文本),无论发送者是否为绑定频道的成员,系统必须不对这些命令消息走决策流水线删除;handler 处理完成后可由 handler 自行选择是否删除命令回显(默认不删)。带尾随文本的非纯调用不被视为命令,必须按普通用户消息走 gating 流水线处理。

#### 场景:非关注者发送 /bind
- **当** 非频道成员 U(非管理员)在群 G 内发送 `/bind`
- **那么** 系统必须走命令分发路径并以权限不足回复,必须不因 U 不是频道成员而删除该命令消息
